# Registre des risques

| ID | Risque | Impact | Probabilité | Plan de mitigation | Statut |
|---|---|---|---|---|---|
| R-001 | Fuite de données entre clients | Élevé | Moyen | permissions objet + tests croisés + `public_id` non prédictibles | Ouvert |
| R-002 | Mauvaise sync Google Drive | Moyen | Moyen | Shared Drive unique + dossier par commande + IDs Drive en base + retries + audit + jobs idempotents + aucun lien brut client | Surveillé |
| R-018 | Mauvaise configuration du compte de service ou du dossier racine Drive | Élevé | Moyen | variables d'environnement obligatoires + `manage.py check` + erreurs de sync sûres côté backend + aucune exposition de secret ou lien brut | Surveillé |
| R-003 | Écart entre runtime local Python `3.10` et runtime cible Docker / CI Python `3.12` | Moyen | Moyen | exécuter les validations de référence dans Docker et CI en `3.12` | Ouvert |
| R-004 | Régression proxy Nginx lors de recréation des conteneurs applicatifs | Moyen | Faible | résolution DNS Docker dynamique + smoke `healthz` après restart | Surveillé |
| R-005 | Audit préparé mais pas encore branché sur les actions sensibles métier | Élevé | Moyen | rendre l’usage de `auditlog` obligatoire dans les prochains lots sensibles | Ouvert |
| R-006 | Frontière exacte des permissions client dans `CustomerMembership` reste volontairement minimale au-delà de `owner` / `member` | Moyen | Moyen | conserver le socle Sprint 01, puis étendre les permissions objet par domaine avant les lots métier sensibles | Surveillé |
| R-007 | Ambiguïté future pour les comptes hybrides client + staff si le changement de contexte n’est pas explicite | Moyen | Faible | garder les routes séparées côté serveur, utiliser une entree auth portail dediee et cadrer la redirection post-login par scope | Surveillé |
| R-008 | Confusion possible entre `/admin/` technique et portail staff applicatif | Moyen | Faible | maintenir `/admin/` comme surface d’administration technique et documenter le portail staff métier séparément | Surveillé |
| R-009 | Audit minimal insuffisant si les prochains lots sensibles ne réutilisent pas le point d’entrée commun | Élevé | Moyen | brancher systématiquement `auditlog` dans les services de mutation sensibles | Ouvert |
| R-010 | Le premier lot catalogue / commande pourrait figer un pricing trop rigide | Moyen | Moyen | conserver un prix unitaire simple et des services métier isolés pour permettre l’évolution vers une grille plus riche | Ouvert |
| R-011 | Une commande minimale exposée sans scoping serveur strict pourrait permettre un accès croisé | Élevé | Moyen | imposer `Customer` comme frontière de tenant, filtrer les querysets, durcir le service d’écriture et tester les accès négatifs entre clients | Surveillé |
| R-012 | Le catalogue initial reste global et non personnalisé par client | Moyen | Faible | conserver les routes client scopees et enrichir seulement si un besoin réel de visibilité ou tarification par client apparaît | Surveillé |
| R-013 | Une permission staff générique pourrait redonner un accès trop large aux futurs domaines sensibles | Élevé | Moyen | conserver `access_staff_portal` comme garde d’entrée et exiger des permissions métier dédiées par domaine | Surveillé |
| R-014 | Un service d’écriture réutilisé hors vue HTTP pourrait contourner le scoping de tenant s’il n’est pas auto-protecteur | Élevé | Moyen | faire vérifier le tenant directement dans les services de mutation critiques et le tester hors couche vue | Surveillé |
| R-015 | Une validation MIME déclarative peut laisser passer un fichier dont le contenu réel ne correspond pas au type annoncé | Moyen | Moyen | appliquer une allowlist serveur stricte maintenant et garder le statut de contrôle simple compatible avec une vérification plus avancée ensuite | Surveillé |
| R-016 | Une fuite d’URL brute de stockage exposerait les fichiers de commande hors contrôle serveur | Élevé | Faible | ne jamais exposer `file.url`, médiatiser les téléchargements par backend et vérifier que le contrôle fichier ne sérialise que des métadonnées sûres | Surveillé |
| R-017 | Un rapport de contrôle trop riche dès Sprint 04 pourrait figer l’évolution vers le DTF avancé | Moyen | Moyen | limiter le modèle au strict nécessaire, conserver des champs extensibles et documenter les statuts simples | Ouvert |
| R-019 | Une transition de workflow non autorisée ou incohérente pourrait placer une commande dans un état atelier invalide | Élevé | Moyen | service central avec refus par défaut, matrice explicite des transitions, permissions dédiées, historique et audit, tests positifs/négatifs | Surveillé |
| R-020 | Un scan atelier mal conçu pourrait divulguer un objet client, contourner le workflow central ou accepter une transition non autorisée | Élevé | Moyen | identifiant opaque unique, routes staff-only, résolution et mutation via service central, refus par défaut, log d’audit, tests de permissions et de non-régression | Surveillé |
| R-021 | Une intégration Sendcloud mal configurée ou trop exposée pourrait divulguer des secrets ou bloquer les expéditions | Élevé | Moyen | variables d’environnement obligatoires + service centralisé + aucun secret en réponse/log + tests d’erreurs API mockées + routes staff-only + audit minimal | Surveillé |
| R-022 | Une création d’expédition répétée ou incohérente pourrait dupliquer label/tracking pour une même commande | Moyen | Moyen | contrainte métier explicite une expédition MVP par commande + garde-fou `ready_to_ship` + refus de doublon + tests de cohérence | Surveillé |
| R-023 | Une UI consolidee client/staff pourrait melanger les scopes et afficher des donnees hors contexte | Élevé | Moyen | routes portail strictement separees (`/client` vs `/staff`), login portail dedie, checks backend de permissions par panneau, tests UI de separation et refus 403 | Surveillé |
| R-024 | Une confirmation/capture PayPal non idempotente pourrait dupliquer les effets metier et la facturation | Élevé | Moyen | endpoint backend interne tokenise + service centralise + garde-fou idempotent sur `paypal_order_id`/`paypal_capture_id` + test de double capture | Surveillé |
| R-025 | Une exposition de donnees paiement/facture hors scope client pourrait provoquer une fuite inter-tenant | Élevé | Moyen | routes client scopees via `customer_public_id` + resolution `Order -> Customer` serveur + tests d'accès croises client A/B + routes staff separees | Surveillé |
| R-026 | Un polish HTMX mal cadre pourrait masquer un refus permission et donner une impression d'action autorisee | Moyen | Moyen | conserver les checks backend par panneau, afficher erreurs explicites, ajouter tests de refus staff granulaire et acces croise client | Surveillé |
| R-027 | Un tunnel frontend premium pourrait contourner le scope client si les etapes utilisent des IDs commandes hors tenant | Élevé | Moyen | imposer `ScopedCustomerMixin` sur toutes les routes checkout, resoudre les commandes via `OrderService.get_customer_order`, et couvrir le cross-tenant en tests UI | Surveillé |
| R-028 | Une landing marketing premium pourrait sur-promettre sans preuve claire et reduire la confiance reelle | Moyen | Moyen | structurer la page autour de preuves concretes, microcopy sobre, CTA explicites et QA UX mobile/desktop | Surveillé |
